# 要件定義見積もり 確認事項への回答

先方よりいただいた確認事項について、以下のとおりご回答いたします。

---

## 1. 成果物の粒度（目次例・サンプル提示）

**ご質問**

成果物について、「現状分析メモ」「業務フロー整理メモ」「連携仕様メモ」等のアウトプットの粒度をより具体的にイメージできるよう、可能であれば成果物の目次例や過去のサンプルなどをご提示いただくことは可能でしょうか？

**回答**

成果物の粒度を明確にするため、現状のリトプラ様の経理業務の案件にて実装している内容のサンプルを送ります。

- フロー図：現状整理用
- プロジェクト計画：前半の内容

---

## 2. 非機能要件の範囲（権限・監査ログ・バックアップ・セキュリティ・性能）

**ご質問**

非機能要件（権限設計、監査ログ、バックアップ/復旧、セキュリティ要件、性能要件など）につきましては、本フェーズの範囲に含まれる認識で問題ございませんでしょうか？

**回答**

権限設定に関しては、要件定義内のスコープとなるともいます。その他の非機能要件に関しては、基本的には、kintone の標準に準拠するものとしています。

---

## 3. 運用設計（棚卸・返品・破損・廃棄・店舗移動・修正フロー等）の深さ

**ご質問**

運用設計（棚卸、返品、破損、廃棄、店舗移動、修正フロー等）について、どの程度まで要件として落とし込む想定か、考え方をご教示いただけますと幸いです。

**回答**

運用設計は、「誰が・いつ・何をするか」と「システムでどう扱うか」を要件レベルで整理するところまでを範囲として想定しています。

- **本フェーズで行うこと**
  - 棚卸・返品・破損・廃棄・店舗移動・データ修正など、主要な運用パターンの洗い出し
  - 各パターンについて「トリガー」「担当」「入力内容」「承認の要否」「他アプリとの連動」を要件として記載
- **開発フェーズに回すもの**
  - 画面の細かい操作手順、帳票のレイアウト、詳細設計に近い部分

---

## 4. freee・スマレジ連携の範囲（調査まで vs 仕様確定まで）

**ご質問**

freee・スマレジ連携につきましては、「調査」までの範囲となるのか、要件定義として「仕様確定」まで含まれるのか、対応範囲を確認させてください。

**回答**

「要件定義として仕様を確定するところまで」をスコープに含める想定です。

- **本フェーズで実施する内容**
  - 連携方式（API/CSV/手動）の選定と理由の整理
  - 連携するデータ項目・方向（kintone → freee／スマレジ 等）の一覧と対応表
  - 連携頻度・タイミング・エラー時の扱いの方針
  - freee・スマレジの API/仕様の確認と、実現可能性・制約の整理
  - 上記を「連携仕様書（たたき台）」にまとめ、レビューまで実施
- **開発フェーズに回すもの**
  - 実際の API 実装・テスト・本番設定

※ freee・スマレジの API 仕様や制約により、追加ヒアリングや検証が必要になった場合は、その時点でご相談させていただきます。

---

## 5. データ移行のスコープ（取込方針・手順・移行テスト）

**ご質問**

データ移行（現行データの取込方針、移行手順、移行テスト等）については、今回のスコープに含まれる想定でしょうか？

**回答**

データ移行については、「方針と要件の整理」までを本フェーズに含め、「手順書の作成・実行・移行テスト」は開発フェーズ以降とする想定です。

※ 「データ移行の実行」は開発フェーズにて実施予定です。

---

## 6. 要件定義フェーズの想定期間・体制・定例会

**ご質問**

要件定義フェーズの想定期間および体制（担当ロール・稼働割合）、また定例会の頻度などについて、共有いただくことは可能でしょうか？

**回答**

以下のとおり想定しています。

- **想定期間**：12週間（※ 見積書の納期に合わせて記載）
- **体制**：PM/リード 1名、要件定義担当 1名、（必要に応じて）その他要員 1名
- **定例会**：週1回、1時間程度（状況に合わせて設定させていただきます）
- **お客様側**：ヒアリング・レビュー・合意のため、週1回程度のご同席を想定

---

## 7. 「基本設計書」の範囲（画面一覧・帳票一覧・項目定義・IF一覧）

**ご質問**

「基本設計書作成」について、画面一覧・帳票一覧・項目定義・IF一覧など、どの範囲まで含まれる想定かを確認させていただけますと助かります。

**回答**

基本設計書には、以下の範囲まで含める想定です。こちらの LiTPLA 様：要件定義書を参考にしてイメージしてください。

- **機能要件一覧**：アプリ（画面）一覧、主要画面の役割・遷移の概要
- **アプリ一覧**：出力する帳票の一覧、用途・出力タイミングの概要
- **項目定義**：アプリのフィールド（項目）一覧、項目名・型・必須/任意・備考（一覧表形式）

---

## 8. 運用・操作マニュアルの作成タイミング

**ご質問**

運用・操作マニュアルにつきましては、開発フェーズ以降で作成する前提で問題ございませんでしょうか？

**回答**

はい、運用・操作マニュアルの作成は開発フェーズ以降を前提としており、本要件定義フェーズのスコープには含めておりません。

要件定義フェーズでは、操作手順の詳細までは確定しないため、マニュアル作成は開発完了に近いタイミング、またはリリース後の運用フェーズで行う想定です。マニュアル作成を別途ご依頼いただく場合は、開発内容が固まった時点でお見積もりいたします。

---
