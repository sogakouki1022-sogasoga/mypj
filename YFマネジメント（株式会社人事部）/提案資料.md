# 株式会社Y.Fマネジメント様 ご提案資料

**kintone を軸にした顧客・案件の部門横断管理**

---

## 1. ご提案の背景

### 1-1. 本提案に至る経緯

- 株式会社人事部様が、Y.Fマネジメント様の業務改善・システム見直しにおいて **kintone の導入・構築フェーズ** を伴走されている。
- 導入・構築を担うパートナーとして、本提案の機会をいただいた。
- 貴社では **2月末ごろまでにシステムを選定** したいとのご意向があり、本資料にて方針・スコープ・概算を整理した。

### 1-2. ご検討企業の概要

| 項目 | 内容 |
|------|------|
| **社名** | 株式会社Y.Fマネジメント 様 |
| **事業概要** | 司法書士・税理士・社労士・不動産調査士など、複数の士業部門を展開するグループ企業 |
| **組織規模** | 約7部門／グループ内 約8法人、総勢約300名（BtoB部門 約60名を先行対象とする想定） |
| **特徴** | グループ＝複数法人の集合。概念としては「一社」として横断管理したいニーズ |

### 1-3. 提案の前提

- 仕組みとしては **CRM（顧客管理）に近い** イメージで、まずは **CRM の形で構築** することを提案する。
- **短期的な目標**（接点情報の蓄積・入力に慣れる）と **中長期的な目標**（売上連携・会計連携）を分けて段階的に進める。
- 社内にシステム専任部門がなく、**ノーコード・低コスト** での改善を重視されている前提で設計する。

## 2. 課題整理と解決の方向性

### 2-1. 現状の課題

| 分類 | 課題 |
|------|------|
| **情報の分断** | 各部門が独立運営のため、顧客情報・営業情報が部門ごとにバラバラ |
| **ツールの混在** | 顧客管理は Excel、一部専用システム等が混在。名刺も部門ごとに個別管理 |
| **属人化** | 売上管理や接点履歴が個人に依存し、グループ全体で「誰がどこと接点を持っているか」が見えない |
| **重複・機会損失** | 他部門ですでに訪問している企業に気づかず訪問するなど、営業資産が活かしきれていない |
| **既存ツール** | マネーフォワード、rakumo、Eight Team 等が点在。将来的な連携を考慮したい |

### 2-2. 目指したい方向性（解決の方向性）

- **グループ横断で「どこの企業の誰と接点があるか」を一覧できる**  
  → 法人番号をキーにした顧客（法人）ベースのデータベースを整備する。
- **接点情報・活動履歴を蓄積し、部門長も確認しながら進められる**  
  → 顧客に紐づいた活動登録・問い合わせ・クレームなどを一元管理する。
- **案件（タスク）の進捗を「いつまでに何をやるか」で可視化する**  
  → 部門ごとに業務は違うが、「タスク管理」の形で統一し、まずは入力と進捗の見える化を実現する。
- **将来的に売上・契約情報や会計（マネーフォワード等）と連携**  
  → 一気にやると混乱するため、まずは接点情報と既存データの連携から始め、会計連携はその後のフェーズとする。

### 2-3. 解決のアプローチ（なぜこの順番か）

1. **まずは「入力することに慣れてもらう」**  
   データの箱（アプリ）を整え、名刺・法人ベース情報・接点情報を蓄積するところまでを第一の目標にする。
2. **CRM としての土台を固めてから案件・日報へ広げる**  
   顧客マスタと接点が軸になってから、タスク管理・日報表示を追加する。
3. **会計連携は第二弾以降**  
   売上情報やマネーフォワード連携は、運用が落ち着いてから検討する。

## 3. システム構成イメージ

### 3-1. 全体像（データの流れ）

```
［名刺管理：Eight Team 等］
        ↓ 取り込み（連携 or CSV）
［法人番号ベースの顧客マスタ］  ← 1企業1キー（法人番号）
        ↓ 部門別に必要な情報を参照
［部門別の顧客・案件情報］
        ＋
［活動管理］［対応管理］  ← 顧客に紐づけて蓄積（活動＝記録、対応＝問い合わせ・クレーム）
        ＋
［タスク管理］  ← 部門ごとにアプリを分けて運用（税理士法人用・社労士法人用など）
        ↓ Phase2
［日報］  ← タスクと紐づけて参照
        ↓ Phase3
［案件管理］［契約管理］［売上管理］  ← 顧客→案件→契約→売上。売上から会計連携
```

- **法人番号** を共通のキーにし、名刺・顧客マスタ・部門別情報・活動・対応・タスク・案件・契約・売上を横串で参照できる形を目指す。
- kintone 上では「1アプリ＝1シート」のようなイメージで、アプリ同士を関連レコード等で紐付け、顧客画面から活動・対応・タスク・案件・契約・売上を参照できるようにする。

### 3-2. kintone内の紐づけ（アプリ間の関連）

kintone 内では、関連レコードやルックアップを用いてアプリ同士を紐づけ、一覧性・追跡性を確保します。

- **顧客マスタ** に紐づくもの：活動管理、対応管理、タスク管理（顧客単位で活動・対応・タスクを参照）
- **案件管理** に紐づくもの：タスク、日報（案件単位でタスク進捗や日報を参照・集計）
- **案件 → 契約 → 売上** の流れ：案件管理に契約管理を紐づけ、契約管理に売上管理を紐づけ、顧客→案件→契約→売上の一連の流れを追えるようにする

これにより、顧客画面から「どの案件で、どんなタスク・日報があるか」を辿れるほか、案件画面から「その案件に紐づくタスク・日報」を一覧・集計できる形を想定しています。

### 3-3. 主要な機能イメージ

| 機能ブロック | 内容 |
|--------------|------|
| **名刺・顧客マスタ** | Eight Team 連携（または CSV 取り込み）→ 法人番号を軸にした顧客マスタ。部門ごとに参照する情報を切り分け。 |
| **活動管理** | 訪問・電話・メール・商談メモなど「誰が・いつ・何をしたか」を記録。顧客に紐づけて蓄積。 |
| **対応管理** | 問い合わせ・クレームを1アプリで種別で切り分け。対応期限・担当・ステータス・内容を管理。 |
| **タスク管理** | 「いつまでに何を完了するか」を管理。部門ごとにアプリを分け（例：税理士法人用・社労士法人用）、混在を防ぎつつ運用。 |
| **案件管理・契約管理・売上管理**（Phase3） | 案件（見込み・進行中）→ 契約（締結済み）→ 売上（請求・入金）。売上管理からマネーフォワードと連携。 |
| **権限設計** | 複数法人・部門を考慮したシンプルなアクセス権（グループや組織で見える範囲を制御）。 |

### 3-4. 既存ツールとの関係

- **Eight Team**：名刺データの取り込み起点。kintone 連携 or CSV で顧客マスタに反映する想定。
- **マネーフォワード**：Phase3 で売上管理アプリから請求・入金データを連携。API の利用可否は契約プラン次第のため要確認。
- その他（rakumo 等）：必要に応じ、手作業でのデータ取り込みや CSV 連携で「箱」に格納するイメージ。

## 4. 導入フェーズとスコープ

### 4-1. フェーズの考え方

- **Phase1**：顧客管理周辺（CRM の土台）を構築し、入力と蓄積に慣れていただく。
- **Phase2**：タスク管理・日報表示を追加し、部門横断で「誰が何をしているか」を見える化する。
- **Phase3**：案件管理・契約管理・売上管理の3アプリを構築し、売上管理からマネーフォワード連携を検討する（要望・予算に応じて実施）。

### 4-2. フェーズ別スコープ

| フェーズ | スコープ | 主な成果物・ゴール |
|----------|----------|----------------------|
| **Phase1** | 顧客管理周辺（CRM） | 名刺連携→顧客マスタ（法人番号ベース）→部門別情報→活動管理・対応管理の蓄積。初期データ整備支援・研修まで。 |
| **Phase2** | タスク管理・日報 | 部門別タスク管理アプリの構築、日報連携・表示、権限設計。テスト・納品・研修。 |
| **Phase3** | 案件・契約・売上・会計連携 | 案件管理・契約管理・売上管理アプリの構築、売上管理からマネーフォワード連携（要確認）。範囲は要件確定後に確定。 |

### 4-3. スコープの境界（含む・含まない）

- **Phase1 に含める**：要件定義・設計、名刺連携（Eight 想定）、顧客マスタ・部門別情報・活動管理・対応管理アプリ、初期データ整備支援、テスト・納品・研修。
- **Phase2 に含める**：タスク管理アプリ（部門別分割）、日報連携・表示、権限設計、テスト・納品・研修。
- **Phase3 に含める**：案件管理・契約管理・売上管理アプリの構築、売上管理からマネーフォワード連携（要確認）。要件確定後に範囲を確定。
- **保守・運用支援**：軽微な改修は別途契約。運用全体に関わる整備も別途相談。
- **会計連携**：Phase3 で売上管理アプリからマネーフォワードへ連携。API 等の可否は契約確認後に確定。
