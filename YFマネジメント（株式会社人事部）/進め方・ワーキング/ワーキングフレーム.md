# 株式会社Y.Fマネジメント様 kintone 導入 ワーキングフレーム

**提案資料に基づく作業範囲・成果物・役割・スケジュールの整理**

---

## 1. プロジェクトの目的とゴール

### 1-1. 目的

- kintone を軸に **顧客・案件・活動履歴の部門横断管理** を実現する。
- 法人番号をキーにした **顧客マスタ** を整備し、名刺・接点・タスクを蓄積・参照できる土台を構築する。
- まずは **入力に慣れていただく** ところまでを Phase1 のゴールとし、段階的に Phase2・Phase3 へ拡張する。

### 1-2. フェーズ別ゴール（完了の定義）

| フェーズ | ゴール（完了とみなす条件） |
|----------|----------------------------|
| **Phase1** | 名刺連携（or CSV取込）→顧客マスタ→部門別情報・活動管理・対応管理が運用可能。初期データ整備と研修を完了し、貴社にて入力・参照を開始できる状態。 |
| **Phase2** | 部門別タスク管理アプリ・日報連携・表示・権限設計が完了。テスト・納品・研修を完了し、運用開始できる状態。 |
| **Phase3** | 案件管理・契約管理・売上管理アプリの構築および売上管理からマネーフォワード連携（要確認）の設計・構築・テスト・納品が完了。 |

## 2. 作業範囲（イン・アウト）

### 2-1. 作業範囲に含む（イン）

| フェーズ | 作業内容 |
|----------|----------|
| **Phase1** | 要件定義・設計（顧客マスタ・法人番号ベース）、名刺管理アプリ構築（Eight連携 or CSV取込想定）、顧客マスタ・部門別情報アプリ、活動管理・対応管理アプリ・関連付け、初期データ整備支援、テスト・納品・研修 |
| **Phase2** | 要件定義・設計（タスク・日報）、タスク管理アプリ構築（部門別分割）、日報連携・表示、権限設計・設定、テスト・納品・研修 |
| **Phase3** | 要件ヒアリング・設計、案件管理・契約管理・売上管理アプリ構築、売上管理からマネーフォワード連携（API可否確認後）、テスト・納品 |

### 2-2. 作業範囲に含まない（アウト）

| 分類 | 内容 |
|------|------|
| **保守・運用** | 納品後の月次保守・軽微な改修は別途契約。運用全体に関わる業務フロー整備・教育の継続も別途。 |
| **他システム開発** | Eight Team・マネーフォワード等の本体機能の改修・カスタマイズ。 |
| **データ入力作業** | 初期データ整備は「支援」（フォーマット整備・取込手順・一部代行の範囲は別途協議）。継続的なデータ入力は貴社側の作業。 |
| **Phase3 の詳細** | マネーフォワード API 利用可否は契約確認後に確定。範囲・工数は要件確定後に見直し。 |

### 2-3. kintone内の紐づけ（アプリ間の関連）

kintone 内では、関連レコードやルックアップを用いてアプリ同士を紐づけます。

- **顧客マスタ** に紐づく：活動管理、対応管理、タスク管理
- **案件管理** に紐づく：タスク、日報（案件単位でタスク進捗・日報を参照・集計）
- **案件 → 契約 → 売上**：案件管理に契約管理を紐づけ、契約管理に売上管理を紐づけ、顧客→案件→契約→売上の流れを追えるようにする

### 2-4. 要確認・前提に依存する範囲

- **名刺連携**：Eight Team 連携の仕様・API/CSV の可否。連携不可の場合は CSV 取込で代替。
- **マネーフォワード**：Phase3 で連携する場合、契約プラン・API 提供の有無を事前に確認する。
- **部門数・アプリ数**：部門別タスク管理アプリの数は、最終的な部門構成に応じて増減する可能性あり。

## 6. 前提条件・制約

### 6-1. 前提条件

- kintone の契約・環境は **貴社にて準備** いただく（サイボウズとの契約、サブドメイン等）。
- **Eight Team** の利用状況・名刺データの出力可否が確認できること。連携不可の場合は CSV 取込で対応可能であることを前提とする。
- 部門別の **窓口・確認いただける方** がいること（要件・画面・権限の確認）。
- Phase3 でマネーフォワード連携を行う場合、**マネーフォワードの契約プラン・API 利用可否** を貴社にて確認いただくこと。

### 6-2. 制約

- **ノーコード・標準機能を優先** し、JavaScript カスタマイズは必要最小限とする。
- 部門数・アプリ数が大きく増えた場合、**工数・金額は見直し** の対象とする。
- 納品後の **仕様変更・追加開発** は別途見積もりとする。軽微な文言・項目の修正は別途契約（保守）の範囲で対応可能な場合あり。

## 7. リスク・留意事項

| リスク・留意事項 | 対応方針 |
|------------------|----------|
| 名刺連携が想定どおりにできない | 事前に Eight 側の仕様・API/エクスポートを確認。CSV 取込で代替可能であることを共有。 |
| 部門ごとの要件のばらつき | Phase1 では共通土台に絞り、Phase2 のタスク管理で部門別アプリ分割により吸収。要件定義で優先度を明確化。 |
| マネーフォワード API が使えない | Phase3 前に契約確認。API 不可の場合は CSV 取込・手動登録の組み合わせで範囲を再定義。 |
| 初期データの量・品質 | データ整備支援の範囲（工数）を契約時に明示。大量の手入力は貴社側で実施いただく前提でフォーマット・手順を提供。 |

## 8. 本ワーキングフレームの位置づけ

- 本ドキュメントは **提案資料に基づく作業の枠組み** を整理したものであり、契約時の **SOW（作業範囲記述書）や業務委託契約の別紙** のたたき台として利用可能です。
- 契約締結時に、**正式なスコープ・成果物・役割・スケジュール** は本フレームをベースに確定し、必要に応じて契約書・別紙に反映してください。
- プロジェクト着手後、**キックオフ時** に本フレームを共有し、認識合わせの材料として利用することを推奨します。
